# Development

Contributor guide for the Paper repository. For contribution policy
(versioning, changelog rules) see [CONTRIBUTING.md](CONTRIBUTING.md); for
cutting a release see [RELEASING.md](RELEASING.md).

## Prerequisites

- **Go 1.26.4+**
- **golangci-lint v2.11.3** and **mockery v2.53.6** — install everything with:

  ```bash
  make install
  ```

  Tools land in `$(go env GOPATH)/bin`; make sure that directory is on your
  `PATH`. Optional extras: `ripgrep` (faster mock lint; plain grep works too)
  and `docsify-cli` via npm (local docs site).

## Module layout

The repository is a Go workspace (`go.work`, committed) with three modules:

| Module | Purpose |
| --- | --- |
| `.` (root) | The library: root API, `pkg/`, `internal/` (including the embedded PDF engine in `internal/pdf`) |
| `examples` | Larger demo programs (`examples/cmd/...`) |
| `docs` | Docsify site + runnable documentation examples (`docs/assets/examples/...`) |

`examples` and `docs` are nested modules so their dependencies never leak into
consumers of the library. Their `go.mod` files pin a released version of the
root module; the committed `go.work` makes local builds and CI resolve the
root module from source instead. The pins only matter at release time — see
RELEASING.md for the `GOWORK=off` verification step.

`pkg/test` is a regular package of the root module (a dependency-free
re-export of `internal/test`), not a separate module.

### Dependency policy

The root module must stay free of third-party **test** dependencies — that is
why `internal/assert`, `internal/require`, `internal/mocktest`, and
`internal/goleak` exist instead of testify/goleak. See
[internal/README.md](internal/README.md). Do not add new runtime dependencies
without prior discussion.

## Everyday loop

```bash
make dod        # definition of done: build + test + fmt + lint
make build      # compile root + examples
make test       # root, examples, and docs-example tests
make fmt        # gofmt + gofumpt + goimports
make lint       # golangci-lint across all modules + mock pattern check
```

## Tests

- **Golden-structure tests:** most component tests build a document, call
  `GetStructure()`, and compare against a JSON snapshot in `test/paper/`
  (via `internal/test`, or `pkg/test` for consumers). When you intentionally
  change document structure, update the matching JSON fixture and inspect the
  diff carefully — fixtures are the API contract for layout.
- **Benchmarks:** `paper_benchmark_test.go`; PRs get an informational
  benchstat comparison via `.github/workflows/benchmark.yml`.

## Mocks

Mocks are generated by mockery into `internal/mocks/` and rewritten to use the
vendored `internal/mocktest` runtime:

```bash
make mocks      # regenerate (mockery v2.53.6 + internal/cmd/mockfix + fmt)
```

Tests must create mocks via constructors (`m := mocks.NewProvider(t)`);
`make lint` enforces this through `shell/mock-check.sh`.

## Docs site

```bash
make docs       # docsify serve docs/ on http://localhost:3000
```

`docs/plans/` holds internal planning documents and is excluded from version
control and the site.
